# Apps Hub 发布操作文档（维护者手册）

## 1. 文档目的

本文档面向本仓库的维护者，说明如何对 **Apps Hub** 站点进行内容更新、版本发布和部署。

本仓库的定位是：

- 作为所有应用的统一官网入口
- 提供各应用的 landing page
- 提供下载页、更新日志、FAQ、隐私政策、条款等页面
- 支持 **简体中文** 与 **English** 双语切换
- 部署到 **GitHub Pages**

---

## 2. 仓库定位与基本原则

### 2.1 仓库职责
本仓库只负责：

- 官网页面
- 多语言内容
- 产品介绍
- 下载入口
- 更新日志展示
- FAQ / Privacy / Terms
- GitHub Pages 静态部署

本仓库通常 **不负责**：

- 桌面应用源码开发
- Tauri 打包
- Apple 签名 / notarization
- Mac App Store 提交
- 桌面应用二进制构建

### 2.2 发布原则
1. 站点必须保持可静态构建与可静态部署。
2. 任何页面改动都必须同时考虑中文和英文版本。
3. 所有 app 内容必须可扩展，不能只为单个 app 写死。
4. 发布前必须检查关键页面的中英文一致性。
5. 下载链接、版本号、更新日志必须与实际发布状态一致。

---

## 3. 技术栈与部署方式

### 3.1 技术栈
- Astro
- TypeScript
- Markdown / Content Collections（如项目实现中采用）
- GitHub Actions
- GitHub Pages

### 3.2 部署方式
建议采用：

- GitHub Pages
- GitHub Actions 作为部署来源

不要依赖手工把构建结果推到某个分支，除非仓库已经明确采用该方案且维护成本可接受。

---

## 4. 目录与内容维护约定

以下为推荐约定；如果实际项目稍有不同，以仓库现状为准，但原则不变。

## 4.1 页面结构
建议至少包含：

- 首页
- 应用列表页
- 单个应用 landing page
- 单个应用下载页
- 更新日志页
- FAQ
- Privacy Policy
- Terms of Service

## 4.2 多语言结构
必须支持：

- 简体中文
- English

推荐路由形式：

- `/zh-cn/`
- `/en/`

以及：

- `/zh-cn/apps/all-my-ai/`
- `/en/apps/all-my-ai/`

### 4.3 内容修改原则
任何页面内容修改时，必须同时检查：

- 中文版本是否更新
- 英文版本是否更新
- 页面标题是否同步
- SEO 文案是否同步
- 下载链接是否同步

如果暂时只完成一种语言，也必须在提交说明中明确标记另一语言尚未同步，不得默默遗漏。

---

## 5. 维护者的常见任务

### 5.1 新增一个 app
当需要新增一个 app 页面时，通常需要：

1. 增加该 app 的中英文内容
2. 增加该 app 的 landing page
3. 增加该 app 的下载页
4. 增加该 app 的 changelog / release notes 入口
5. 把该 app 加入首页或应用列表页
6. 检查语言切换是否能在该 app 页面正确跳转
7. 检查导航、页脚、SEO 信息是否完整

### 5.2 修改已有 app 的文案
需要检查：

- 中英文内容是否同步
- 页面中是否有写死旧产品名
- 下载按钮是否仍然指向正确位置
- 版本号 / 发布时间是否一致

### 5.3 更新下载入口
如果某个 app 的下载地址变化：

1. 修改中文下载页
2. 修改英文下载页
3. 修改可能出现在首页中的 CTA
4. 修改 FAQ 中相关说明
5. 检查是否有 old link 仍残留在其他页面

### 5.4 发布更新日志
每次 app 版本发布后，应当同步更新：

- 对应 app 的 changelog / release notes
- 下载页中的当前版本信息（如果站点展示版本）
- 首页中的版本展示（如果有）

## 6. 本地开发流程

以下命令以常见 Node.js 项目为例，具体以仓库内 `package.json` 为准。

### 6.1 安装依赖
```bash
npm install
```

如果仓库明确使用 `pnpm` 或 `yarn`，则使用仓库规定的包管理器，不要混用。

### 6.2 启动本地开发
```bash
npm run dev
```

启动后检查：

- 首页是否正常打开
- 中英文语言切换是否可用
- app 页面是否可访问
- 静态资源是否正常加载

### 6.3 本地构建
```bash
npm run build
```

构建必须成功，且不能只在 dev 模式可用。

### 6.4 本地预览
如果项目提供预览命令，建议在发布前执行：

```bash
npm run preview
```

## 7. 发布前检查清单

每次准备合并到主分支或发布前，维护者至少应检查以下内容。

### 7.1 页面与内容检查
- [ ] 首页正常显示
- [ ] 应用列表页正常显示
- [ ] 关键 app 页面可访问
- [ ] 下载页可访问
- [ ] changelog 可访问
- [ ] FAQ 可访问
- [ ] Privacy / Terms 可访问

### 7.2 多语言检查
- [ ] 中文首页正常
- [ ] 英文首页正常
- [ ] 中文 app 页面正常
- [ ] 英文 app 页面正常
- [ ] 中文下载页正常
- [ ] 英文下载页正常
- [ ] 语言切换按钮可用
- [ ] 切换语言后尽量落在对应页面，而不是总跳首页

### 7.3 链接检查
- [ ] 下载链接正确
- [ ] 社交/联系链接正确
- [ ] changelog 链接正确
- [ ] FAQ / Privacy / Terms 链接正确
- [ ] 没有明显 404

### 7.4 构建检查
- [ ] `npm run build` 成功
- [ ] 没有明显 TypeScript 错误
- [ ] 没有明显路由冲突
- [ ] 没有构建时缺失环境变量导致失败

### 7.5 页面质量检查
- [ ] 桌面端显示正常
- [ ] 移动端显示基本正常
- [ ] 无明显排版错乱
- [ ] 中英文标题和按钮没有留空
- [ ] 不存在明显占位文案（如 TODO、TBD、Lorem ipsum）

## 8. GitHub 发布与部署流程

## 8.1 推荐分支策略
建议：

- `main`：可部署主分支
- 功能修改通过分支或 PR 进入 `main`

### 8.2 推荐流程
1. 在功能分支完成修改
2. 本地运行：
   - `npm install`
   - `npm run build`
3. 自检中英文页面
4. 提交 PR
5. 审核通过后合并到 `main`
6. GitHub Actions 自动部署到 GitHub Pages
7. 合并后检查线上站点

### 8.3 GitHub Pages 配置建议
建议使用：

- Pages source: **GitHub Actions**

需要在 GitHub 仓库设置中确认：

- Pages 已开启
- 部署来源正确
- 自定义域名（如果已配置）正常
- HTTPS 已开启

### 8.4 自定义域名
如果仓库绑定了自定义域名，需要额外确认：

- DNS 解析正确
- GitHub Pages 中配置的 Custom domain 正确
- HTTPS 正常启用
- 域名未过期

## 9. 更新日志维护规范

## 9.1 更新日志应包含的内容
每条 release note 建议包含：

- 版本号
- 发布日期
- 新增内容
- 修复内容
- 兼容性说明（如需要）
- 下载变化（如需要）

## 9.2 多语言要求
如果站点对外展示更新日志，则：

- 中文用户可看到中文更新说明
- 英文用户可看到英文更新说明

可以采用：
- 一份中英文分别维护的 changelog
- 或同一条 release note 内含双语内容

但整体风格必须统一。

---

## 10. 下载页维护规范

下载页建议至少包含：

- 应用名称
- 当前版本（如适用）
- 支持平台
- 下载按钮
- 安装说明（如有）
- 常见问题入口
- 更新日志入口

如果某个 app 暂未开放下载，应明确写：

- 即将发布
- 加入等待列表
- 或仅提供产品介绍

不要放一个无效下载按钮。

---

## 11. 维护者不要做的事情

- 不要只更新中文不更新英文，除非明确标注
- 不要把某个 app 的写死逻辑复制到全站公共组件里
- 不要把下载链接散落硬编码在多个页面而不做统一管理
- 不要未经验证直接修改域名、Pages 配置或部署流程
- 不要把测试中的占位文案留在生产页面
- 不要把并未公开的产品提前加入正式导航
- 不要把桌面应用源码或敏感发布信息混入本 public 仓库

## 12. 建议的维护流程模板

### 小改动（文案、链接、FAQ）
适用于：
- 文字修正
- 下载链接更新
- FAQ 补充

流程：
1. 新建分支
2. 修改中英文内容
3. 本地运行与构建
4. 检查关键页面
5. 提交 PR
6. 合并并观察线上结果

### 中等改动（新增一个页面或 app）
适用于：
- 新 app landing page
- 新下载页
- 新 changelog 模块

流程：
1. 设计信息结构
2. 增加中英文内容
3. 增加对应路由/页面
4. 检查语言切换
5. 本地 build
6. 走 PR 审核
7. 合并后验证线上

### 大改动（导航重构、样式重构、部署改造）
适用于：
- 站点首页重做
- 整体 i18n 结构调整
- Pages / Actions 部署调整

流程：
1. 先提交设计方案
2. 明确影响范围
3. 在独立分支完成
4. 提供本地截图或预览
5. 经审阅后再合并
6. 合并后重点验证线上可用性

## 13. 常见故障排查

### 13.1 本地能跑，线上 404
检查：
- Astro 配置中的 `site` / `base`
- GitHub Pages 部署路径
- 是否使用了错误的 repo base path
- 多语言路由是否生成完整

### 13.2 语言切换后跳到错误页面
检查：
- 中英文页面 slug 是否一致
- 语言映射逻辑是否写死
- 某些页面是否只有一种语言版本

### 13.3 GitHub Pages 部署成功，但样式丢失
检查：
- 静态资源路径是否正确
- 是否写了以 `/` 开头但未考虑 base path 的路径
- 构建输出是否与 Pages 配置一致

### 13.4 某个下载按钮失效
检查：
- 链接是否过期
- 是否仍指向测试地址
- 是否只修改了一种语言页面
- 是否该链接本应从统一配置读取但被手工写死

## 15. 最小发布 SOP（维护者速查版）

每次准备发布站点改动时，按下面执行：

1. 拉取最新代码
2. 安装依赖
3. 修改内容或页面
4. 同步中文和英文
5. 执行 `npm run build`
6. 本地检查：
   - 首页
   - app 页面
   - 下载页
   - changelog
   - FAQ
   - Privacy / Terms
7. 检查语言切换
8. 提交 PR
9. 合并到 `main`
10. 等待 GitHub Actions 部署
11. 检查线上页面与链接

- 统一下载链接配置中心

## 17. 最后一条要求

任何会影响用户访问、下载、语言切换或主导航的改动，都不应只做“代码提交成功”层面的验证，必须至少做一次实际页面访问验证。
