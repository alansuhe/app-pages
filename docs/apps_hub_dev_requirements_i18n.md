# Apps Hub 开发要求（给 Codex / Claude Code / Cursor Agent）

你正在为一个新的 **public repo** 开发网站。这个仓库将作为作者所有桌面应用的统一：

- Landing page / 官网入口
- Release / 更新说明中心
- 下载入口
- FAQ / 隐私政策 / 条款页面入口

该站点当前优先服务 macOS 桌面应用，尤其包括：

- `All My AI`

本仓库本身 **不是桌面应用源码仓库**，而是一个独立的官网与发布中心仓库。

---

## 1. 项目目标

请基于以下目标开发：

### 核心目标
1. 为多个 app 提供统一的品牌首页。
2. 为每个 app 提供独立的产品介绍页面。
3. 为每个 app 提供独立的下载页。
4. 为每个 app 提供 changelog / release notes 展示页。
5. 为后续接入 GitHub Releases / 站外更新说明预留结构。
6. 适配 GitHub Pages 部署。
7. 全站以静态站为主，不依赖后端服务。

### 当前阶段目标
当前先重点支持：
- 一个首页
- 一个 `All My AI` 产品页
- 一个 `All My AI` 下载页
- 一个 changelog / updates 页面
- 一个 FAQ 页面
- 一个 Privacy 页面
- 一个 Terms 页面

---

## 2. 技术栈要求

### 必选技术栈
- **Astro**
- **TypeScript**
- **Markdown content collections**（用于 app 内容、changelog、FAQ 等内容管理）
- 纯静态输出（适配 GitHub Pages）

### 可选技术栈
- 仅在必要时使用少量 React 组件
- 样式可使用以下任一方式，但必须保持简单、稳定、易维护：
  - 原生 CSS / CSS Modules
  - Tailwind CSS

### 不要使用
- 不要引入数据库
- 不要引入服务端 API 依赖
- 不要默认使用 SSR
- 不要为了简单 landing page 站点引入复杂全栈架构

---

## 3. 部署目标

该仓库目标部署到：

- **GitHub Pages**
- 使用 **GitHub Actions** 进行构建与部署
- 后续可能绑定自定义域名

实现时必须：
- 使用静态输出模式
- 保证构建产物适合 GitHub Pages
- 预留 `site` 配置位置，方便后续替换为正式域名
- 不要硬编码假域名；如果域名未知，请使用明确 TODO 注释

---

## 4. 信息架构要求

请按照“多应用统一中心”的方式设计，而不是只为单一 app 写死结构。

### 顶层页面
至少需要支持以下页面结构：

- `/` 首页
- `/apps/all-my-ai/` 单应用介绍页
- `/download/all-my-ai/` 单应用下载页
- `/changelog/` 或 `/updates/` 更新中心
- `/faq/`
- `/privacy/`
- `/terms/`

### 多应用扩展要求
后续应该可以很容易新增：
- `/apps/<slug>/`
- `/download/<slug>/`
- 对应 changelog 内容

不要把 `All My AI` 写死到全站架构里；应让它成为第一个 app 实例。

---

## 5. 内容组织要求

请优先采用“内容与页面分离”的方式。

### 推荐内容结构
建议使用 Astro content collections 或等价结构，内容大致可分为：

- `apps`：每个应用的描述信息
- `changelog`：版本更新记录
- `faq`：常见问题
- 可选：`legal` 或直接使用页面文件承载法务内容

### app 内容至少应支持这些字段
每个 app 的内容模型建议至少包含：

- `title`
- `slug`
- `tagline`
- `description`
- `platforms`
- `status`
- `primaryCta`
- `downloadPath`
- `icon`（若暂时没有，可留空或预留）
- `heroImage`（若暂时没有，可留空或预留）

### changelog 内容至少应支持这些字段
- `app`
- `version`
- `date`
- `summary`
- `published`

### FAQ 内容至少应支持这些字段
- `question`
- `answer`
- `order`
- `published`

不要把主要内容硬编码在组件内部。

## 6. UI / UX 要求

整体风格要求：
- 简洁
- 现代
- 轻量
- 偏产品官网，而不是文档站模板感太重

### 首页应包含
- 站点标题/品牌说明
- 所有 app 的列表或卡片
- 每个 app 的一句话介绍
- 跳转到 app 详情页的入口

### 单应用页应包含
- app 名称
- tagline
- 核心卖点
- 平台信息（当前以 macOS 为主）
- 下载入口
- changelog 入口
- FAQ / 隐私 / 条款相关入口

### 下载页应包含
- 当前下载方式说明
- 平台支持说明
- 版本说明占位
- 发布渠道说明占位
- 明确区分：
  - 站外 Direct 下载
  - Mac App Store（如果未来启用）

注意：
- 当前不要求真实接入下载 API
- 可以先使用占位链接或基于配置的链接
- 不要伪造真实下载地址

### 更新页应包含
- 按 app 聚合或筛选的 changelog 列表
- 单条更新的标题、版本、日期、摘要

## 6.1 多语言要求（必须支持中英文切换）

本项目必须支持：
- **简体中文**
- **English**

这不是可选增强项，而是当前阶段的硬性要求。

### 多语言实现要求
- 必须提供可用的语言切换 UI。
- 必须保证首页、app 页面、下载页、changelog 页面、FAQ、Privacy、Terms 至少具备中英文版本或中英文内容渲染能力。
- 不要只翻译导航；主要页面内容也必须支持中英文。
- 语言切换后，页面标题、描述、导航、按钮文案、页脚文案都应同步切换。

### 推荐实现方式
优先采用以下任一方案：
1. 基于 Astro i18n 路由能力或等价静态路由方案实现。
2. 基于内容集合按语言分别组织内容。
3. 使用清晰的字典/翻译资源文件管理共享 UI 文案。

### 路由建议
推荐采用带语言前缀的静态路由结构，例如：
- `/zh-cn/`
- `/en/`
- `/zh-cn/apps/all-my-ai/`
- `/en/apps/all-my-ai/`

如果你采用其他等价方案，也必须满足：
- 路由清晰稳定
- GitHub Pages 可正常静态部署
- 后续新增 app 和新增语言时改动成本低

### 默认语言与切换要求
- 默认语言建议为 **English** 或可配置默认语言，但实现中必须明确写出默认策略。
- Header 中应提供清晰的语言切换入口。
- 页面切换语言时，应尽量跳转到当前页面对应语言版本，而不是始终跳回首页。

### 内容组织要求
多语言内容不能全部硬编码在页面组件里。建议拆分为：
- 共享 UI 翻译字典
- 按语言区分的 app 内容
- 按语言区分的 changelog / FAQ / 法务内容（如果当前阶段不完整，可先对部分内容做占位，但结构必须支持）

### 开发约束
- 不要临时用 scattered string replacement 的方式拼凑多语言。
- 不要把中文和英文大段文案直接硬编码在多个页面组件中重复维护。
- 必须让后续新增语言具备可扩展性。
